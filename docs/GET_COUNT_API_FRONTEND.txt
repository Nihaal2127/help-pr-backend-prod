================================================================================
POST /api/getCount — dashboard aggregate counts (frontend reference)
================================================================================

ENDPOINT
  Method:   POST
  Path:     /api/getCount
  Base URL: your API host + stage (e.g. …/staging) — full path is /api/getCount

AUTHENTICATION
  Required: Bearer JWT in header
    Authorization: Bearer <access_token>

CONTENT-TYPE
  application/json

--------------------------------------------------------------------------------
REQUEST BODY (JSON)
--------------------------------------------------------------------------------

REQUIRED
  type (string | number)
    Which dashboard module’s counts to return. Use one of the string keys below,
    or the numeric code if you already map them in the app.

OPTIONAL — franchise scope
  franchise (string, preferred)
    MongoDB ObjectId string (24 hex chars). When sent, counts are scoped to that
    franchise where the backend supports it (see table below).

  franchise_id (string, fallback)
    Same as franchise. If both are non-empty, "franchise" wins.

  Rules when franchise / franchise_id is present:
    • Must be a valid ObjectId → otherwise HTTP 409 "Invalid franchise id."
    • User must be allowed to read that franchise → otherwise HTTP 403/404
      (super admin 5 and staff 6: any franchise; franchise admin 1: own
       franchise or franchises they admin; employee 3: only own franchise_id).

--------------------------------------------------------------------------------
"type" VALUES (string keys → same as internal numeric code)
--------------------------------------------------------------------------------

  location-management           1
  service-management            2
  user-management                 3
  financials                    4   (aliases: order-payment, financial-order-payments → 4)
  order-payment                 4
  financial-order-payments      4
  franchise-management          6
  expenses                      7
  settings-expense-categories   8
  content-management            9
  my-franchise                  10  (alias: my_franchise)
  quote-management              11  (aliases: quote_management, quotes)
  partner-management            12
  settings-role                 13  (aliases: settings_role, settingsrole)
  order-management              14  (aliases: order_management, orders)
  partner-post-management       16  (aliases: partner_post_management, partner-posts, partner_posts)
  partner-portfolio-management  17  (aliases: partner_portfolio_management, partners-browse, partners_browse)
  settings-offers               15  (aliases: settings_offers, offers-management)

  You may also send "type" as a string of digits (e.g. "11") — it is parsed as a number.

--------------------------------------------------------------------------------
FRANCHISE PARAMETER — WHAT CHANGES PER "type"
--------------------------------------------------------------------------------

  When franchise / franchise_id is OMITTED: behaviour is global or caller-scoped
  depending on "type" (e.g. type 2 service-management uses global Category/Service
  counts; type 2 no longer auto-scopes franchise admins without sending the id).

  When franchise / franchise_id is SENT (and access OK):

  1  location-management
     Counts only that franchise’s state, city, and linked areas.

  2  service-management
     With franchise_id: same category/service fields as my-franchise (global catalogue
     totals, mapping-based active/inactive, requested_* from franchise users) — no
     employees or areas. Aligns with GET franchise-category|service/getAll all_* arrays.
     If franchise is omitted, counts are global Category/Service rows (is_request /
     is_active) for approved catalogue only.

  3  user-management
     User counts filtered by user.franchise_id = that franchise.

  4  financials / order-payment / financial-order-payments
     Aggregates order rows (same scope as GET /api/order/getAll). See
     “FINANCIAL ORDER PAYMENTS (TYPE 4)” and docs/FINANCIAL_ORDER_PAYMENTS_API.md.

  6  franchise-management
     Counts only that franchise row.

  7  expenses
  8  settings-expense-categories
     Filtered by expense / expense category franchise_id.

  9  content-management
     No franchise field on content — counts unchanged; franchise id still
     validated for access if you send it.

  10 my-franchise
      Counts for the resolved franchise scope (employees, areas, catalogue vs
      mapping, pending requests). See section “MY-FRANCHISE (TYPE 10)” below.

  11 quote-management
      Same franchise/role scope as GET /api/quote/getCounts (resolveQuoteListScope).
      Super admin / staff: all quotes, or filter with franchise / franchise_id.
      Franchise admin / employee: always scoped to their franchise.
      Partner / customer: 403.

  12 partner-management
      Subscription plan totals stay global; partner subscription counts only
      partners (type 2) with that franchise_id.

  13 settings-role
      Super admin / staff: optional franchise scope (same id as above).
      Franchise admin / employee: still tied to their own franchise in logic.

  14 order-management
      Same franchise/role scope as GET /api/order/getAll and quote getCounts.
      Super admin / staff: all orders, or filter with franchise / franchise_id.
      Franchise admin / employee: always scoped to their franchise.
      Partner / customer: 403.

  16 partner-post-management
      Same franchise/role scope as GET /api/partner-post/getCounts.
      Super admin / staff: all posts/reports, or filter with franchise / franchise_id.
      Optional partner_id (body or query) narrows to one partner.
      Franchise admin / employee: always scoped to their franchise.
      Partner / customer: 403.

  17 partner-portfolio-management
      Same franchise/role scope as GET /api/partners and GET /api/partners/getCounts.
      Super admin / staff: all eligible partners, or filter with franchise / franchise_id.
      Franchise admin / employee: always scoped to their franchise.
      Partner / customer: 403.

--------------------------------------------------------------------------------
FINANCIAL ORDER PAYMENTS (TYPE 4) — "record" KEYS (four dashboard cards)
--------------------------------------------------------------------------------

  Matches Financial — Order Payments UI. Data source: order collection
  (same as GET /api/order/financial-payments/getAll).

  total_completed_orders       Count where order_status = "completed"
  total_in_progress_orders     Count where order_status = "in_progress"
  total_partner_pending_amount Sum of order.partner_due_amount (₹, rounded to 2 decimals)
  total_user_pending_amount    Sum of order.customer_due_amount (₹, rounded to 2 decimals)

  Example:
    POST /api/getCount
    { "type": "financial-order-payments", "franchise_id": "<ObjectId>" }

--------------------------------------------------------------------------------
ORDER-MANAGEMENT (TYPE 14) — "record" KEYS (four dashboard cards)
--------------------------------------------------------------------------------

  Canonical order_status strings (see enum/order_status_enum.js):

  in_progress   "in-progress"  (legacy DB: 1 or 2; excludes refund payment rollups)
  completed     "completed"    (legacy DB: 3; excludes refund payment rollups)
  cancelled     "cancelled"    (legacy DB: 4; excludes refund payment rollups)
  refunded      order_status "refunded" OR user_payment_status / payment_status
                "refund" | "partially_refund" (includes refund API orders)

  List filter (same buckets): GET /api/order/getAll?order_status=refunded
  Aliases: pending → in-progress; refund → refunded; numeric 1–4 per migration doc.
  Buckets are mutually exclusive (refunded tab wins when payment rollup shows refund).

--------------------------------------------------------------------------------
PARTNER-POST-MANAGEMENT (TYPE 16) — "record" KEYS (six dashboard cards)
--------------------------------------------------------------------------------

  Post status (partner_post.status, deleted_at null):

  published     Live posts — same bucket as GET /api/partner-post/getAll?status=published
  hidden        Admin-hidden posts
  removed       Admin-removed posts

  Report status (partner_post_report.status, parent post in scope):

  pending       Customer reports awaiting review — GET /api/partner-post/reports?status=pending
  reviewed      Reports marked handled
  dismissed     Reports closed with no post action

  Standalone (same buckets):
    GET /api/partner-post/getCounts?franchise_id=<optional>&partner_id=<optional>

  Example:
    POST /api/getCount
    { "type": "partner-post-management", "franchise_id": "<ObjectId>", "partner_id": "<optional>" }

  See docs/PARTNER_POST_ADMIN_FRONTEND.md §5.

--------------------------------------------------------------------------------
PARTNER-PORTFOLIO-MANAGEMENT (TYPE 17) — "record" KEYS (five dashboard cards)
--------------------------------------------------------------------------------

  Eligible partners: same pool as GET /api/partners (verified, active subscription,
  active plan, franchise-scoped).

  total       All eligible partners in scope
  basic       plan_name = basic   (same as GET /api/partners?plan_name=basic)
  silver      plan_name = silver
  gold        plan_name = gold
  platinum    plan_name = platinum

  Standalone (same buckets):
    GET /api/partners/getCounts?franchise_id=<optional>

  Example:
    POST /api/getCount
    { "type": "partner-portfolio-management", "franchise_id": "<ObjectId>" }

--------------------------------------------------------------------------------
MY-FRANCHISE (TYPE 10) — HOW "record" FIELDS ARE COMPUTED
--------------------------------------------------------------------------------

  Franchise scope (which Franchise row(s) are used):
    • Optional body "franchise" / "franchise_id" (after access check), else
      franchise admin (type 1): their franchise_id, or all franchises they admin,
      else any user with franchise_id: that franchise.

  Employees
    total_employee, inactive_employee, active_employee
      Users with type = 3 (employee), deleted_at null, franchise_id in scope;
      inactive/active split on is_active.

  Areas
    total_area, inactive_area, active_area
      Area documents whose _id appears on the franchise’s area_id list(s),
      deleted_at null; inactive = is_active false; active = is_active not false.

  Global catalogue size (approved rows only — is_request: false)
    total_category
      Count of non-deleted Category documents with is_request false.
    total_service
      Count of non-deleted Service documents with is_request false.
      Matches franchise-category|service getAll all_* when pending rows are hidden.

  Franchise mapping vs catalogue (categories)
    active_category
      Sum of array lengths: FranchiseCategory.active_categories for each mapping
      row where franchise_id is in scope (not Category.is_active).
    inactive_category
      max(0, total_category - active_category)
      i.e. global categories minus how many slots are listed as active on the
      franchise mapping (never negative).

  Franchise mapping vs catalogue (services)
    active_service
      Sum of array lengths: FranchiseService.active_services for franchise_id in scope.
    inactive_service
      max(0, total_service - active_service)
      i.e. global services minus active mapping slots (never negative).

  Pending requests (raised by users tied to the franchise)
    requested_category, requested_service
      Global Category / Service rows with is_request true and requested_by in the
      set of user _ids that have franchise_id in the resolved franchise scope.
      Pair with GET /api/category|service/getAll?status=requested* (and for super
      admin / staff also pass franchise_id when viewing one franchise).

--------------------------------------------------------------------------------
SUCCESS RESPONSE (200)
--------------------------------------------------------------------------------

  {
    "success": true,
    "status": 200,
    "record": { ... }   // shape depends on "type" — dashboard card fields only
  }

  Field names in "record" differ per module (e.g. quote-management: new, pending,
  accepted, success, failed). Inspect one successful response per module or ask
  backend for the exact keys.

--------------------------------------------------------------------------------
ERROR RESPONSES (typical)
--------------------------------------------------------------------------------

  400  Invalid or unsupported "type"
  401  Missing/invalid JWT or user not found where applicable
  403  Not allowed to read counts for the given franchise
  404  Franchise not found (when franchise id is used)
  409  Invalid franchise id format
  500  Server error

--------------------------------------------------------------------------------
EXAMPLE REQUESTS
--------------------------------------------------------------------------------

  Global quote dashboard (all franchises, if caller is allowed):
    POST /api/getCount
    { "type": "quote-management" }

  Quote dashboard for one franchise:
    POST /api/getCount
    { "type": "quote-management", "franchise": "<24-char ObjectId>" }

  User management counts for one franchise:
    POST /api/getCount
    { "type": "user-management", "franchise": "<24-char ObjectId>" }

  Same using fallback key:
    { "type": "user-management", "franchise_id": "<24-char ObjectId>" }

--------------------------------------------------------------------------------
RELATED STANDALONE getCounts ENDPOINTS
--------------------------------------------------------------------------------

  GET /api/partners/getCounts?franchise_id=<optional ObjectId>
  Same five buckets as partner-portfolio-management (type 17) on POST /api/getCount.
